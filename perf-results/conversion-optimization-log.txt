  FFI overhead for value export even though the native BigInteger already stores canonical 60-bit limbs in Kotlin.

  through LibTomMath and `toDouble` goes through decimal string formatting.

=== RUN 2 — 2026-04-18 ===
Change: Replaced the remaining conversion-side LibTomMath export calls with limb-native Kotlin code.
- `toByteArray` now writes unsigned big-endian bytes directly from the canonical limb array and applies
  two's-complement fixup locally for negative values.
- `toInt` / `toLong` now read the low 32 / 64 bits directly from the canonical limbs and preserve JVM wraparound semantics.
- `toString(radix)` now formats from Kotlin-owned limbs using grouped division with a `ULong` fast path for small magnitudes.
- `hashCode` now folds the magnitude directly instead of round-tripping through LibTomMath export.
