=== POOL ALLOCATOR INVESTIGATION — 2026-04-16 ===

Tested three pool approaches to eliminate the ~0.16us allocation floor:

1. pthread_mutex pool (cleaner + main thread): REGRESSION — mutex overhead ~0.3us per op
   addMethod small: 0.16 → 0.45 us/op (2.8x slower)

2. Spinlock pool (AtomicInt CAS): REGRESSION — spinlock overhead ~0.15us per op
   addMethod small: 0.16 → 0.33 us/op (2x slower)

3. Local-only pool (no synchronization, freeMp temporaries only): NEUTRAL
   addMethod small: 0.16 → 0.19 us/op (within noise)
   Pool only recycles freeMp temporaries, not BigInteger result allocations.

DIAGNOSTIC: Removed createCleaner to measure its isolated cost:
   addMethod small: 0.16 → 0.093 us/op (without cleaner)
   → createCleaner costs ~0.07us per BigInteger construction (~45% of the floor)
   → nativeHeap.alloc + mp_init + object creation costs ~0.09us (~55%)
   → JVM add small: 0.018 us/op → even without cleaner, native is 5x slower

CONCLUSION: The ~0.16us floor has two components:
  - createCleaner registration: ~0.07us (irreducible with current Kotlin/Native GC API)
  - nativeHeap.alloc + mp_init + BigInteger construction: ~0.09us
  Pool allocator cannot help because:
  - Result mp_ints go through cleaner (finalizer thread) → need synchronization → overhead > savings
  - Temporary mp_ints (freeMp) are a minority of allocations
  Closing the gap to JVM requires changes at the Kotlin/Native runtime level (cheaper GC integration
  or value-type BigInteger without per-instance cleanup).
